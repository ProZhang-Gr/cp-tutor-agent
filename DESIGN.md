# 设计说明 · ARENA 算法竞赛辅导智能体

> 本文讲的是**为什么这样设计、做了哪些取舍**，与《功能报告》（讲"有什么功能"）、README（讲"怎么用"）互补。
> 面向想理解系统骨架与工程判断的读者。

---

## 0. 设计定位：从「演示」到「可信」

一个能跑的多智能体 demo 不难，难的是让懂行的人相信它「不是玩具」。我们认为分水岭是两点，
本轮设计的全部重心都压在这两点上：

1. **判题要有「地面真值（ground truth）」。** 如果 AC 的含义只是"某个会幻觉的模型觉得你对了"，
   那整套判题就是空中楼阁。必须建立确定性的、可复现的判定来源。
2. **「Agent」要名副其实。** 真正的 agent 有工具、有反馈闭环、能自我纠错；
   而不是把几次顺序的 LLM 调用串起来就叫"多智能体"。

其余功能（题库、画像、会员、报告）都是围绕这两个支点的工程纵深。

## 1. 多智能体编排：状态机，而非一条长 prompt

**取舍**：用 LangGraph 的 `StateGraph` 把辅导拆成显式的节点与条件边，而不是写一个巨型 prompt 让模型"一口气全干"。

**理由**
- 每个角色（分析/规划/审查/判题…）有独立的系统提示词与输出契约（JSON schema），职责单一、可单测、可替换；
- **条件分支**是关键：分析流里"不是有效算法题就不硬分析"；评测流里"语法都过不了就跳过判题"。
  这些分支用图的 `add_conditional_edges` 表达，逻辑清晰且可观测；
- `graph.stream(stream_mode="updates")` 天然提供**逐节点的中间产出**，直接驱动前端"智能体逐个点亮"的可视化——
  编排结构与 UI 叙事是同构的。

**两条工作流**
```
分析流  检索 → 分析 →[是有效题?]→ 规划 / 直接结束
评测流  审查 →[语法过?]→ 双轨判题 → 汇总
```
调试回路不放进图里，而是**独立的按需 ReAct 循环**（见 §3），因为它的步数与分支由模型在运行时决定，
不是预先固定的 DAG——这本身就说明了它和"流水线"的区别。

## 2. 判题真值化（核心设计）

### 2.1 问题
competitive programming 的判题需要**确定性的期望输出**。原型阶段是"让 LLM 生成测试用例并想象期望输出"，
这有一个致命缺陷：**裁判本身会幻觉**，AC 不可信。

### 2.2 为什么不直接"接 OJ"
我们考察过 Codeforces / LeetCode / 洛谷 / AtCoder：
- **题面与元数据**（标签、难度分）多数可得；
- **但隐藏测试数据几乎都不公开**。Codeforces 有官方 API 却不给 system tests；LeetCode 判题在它自家服务器。

也就是说，"把自己做成一个真 OJ"这条路走不通——你永远缺真值。

### 2.3 双轨判题（`core/judge.py`）
按题目是否自带官方数据，自动选轨：

| 轨道 | 触发条件 | 真值来源 | 强度 |
|------|----------|----------|------|
| **真值判定 truth** | 题目带官方测试数据 | `data/tests/<id>.json` 里的真实数据 | 确定性，等同 OJ |
| **差分对拍 stress** | 用户自己粘的题、无数据 | LLM 写的**朴素暴力解** | 经验性，但能抓出绝大多数 bug |

- 真值轨：100 道导入真题全部带官方数据，逐组比对，遇首个失败即停并给反例；
- 对拍轨：让 LLM 产出 `brute_code`（朴素正确解）+ `gen_code`（小规模随机数据生成器），
  **先用题面样例校验暴力解可信度**（暴力解连样例都过不了就降可信度标注），
  再用大量小随机数据让用户解与暴力解对拍，**第一处分歧即最小反例**。

**为什么对拍是对的补丁**：它正是竞赛选手手动「对拍」的自动化版本。即便接入了 Codeforces 也拿不到隐藏数据，
对拍恰好**用一个"显然正确但慢"的解当真值**，补上了这块缺口——而且小规模随机数据天然产出**最小反例**，
直接喂给下游的调试 agent。

### 2.4 得分设计：通过用例占比
原型曾用"判题裁决主导 + 审查分加成"（真值 AC≈95、对拍 AC≈85…），但它有解释成本：基准分因轨道而异、"明明 AC 却不是满分"都得额外说明。最终回归最朴素、对学生最可解释的口径：

- **得分 = 通过用例数 ÷ 总用例数 × 100** —— "过了几成给几成"，一眼看懂；
- **CE（语法/编译错误）记 0** —— 代码根本没跑起来；
- **无可判用例**（既无官方数据、又无法自动对拍）时，退回**静态审查分**兜底。

判题裁决（AC/WA/TLE/RE/CE）与"真值/对拍"轨道仍照常展示，只是不再折算成一个需要解释的加权分。**可解释性优先于"看起来高级"**——学生该清楚地知道自己为什么是这个分。

## 3. Agentic 调试回路（核心设计）

### 3.1 协议
`core/debugger.py` 手写了一个 **ReAct 循环**（而非黑盒地交给某框架），便于完全掌控与流式呈现：

- **唯一工具** `run(code, stdin)`：在沙箱里真实执行一段 Python，返回 stdout/stderr/状态；
- 每一步模型输出严格 JSON：`{thought, action: run|finish, code, stdin, conclusion}`；
- `action=run` → 我们执行并把观察结果作为下一轮的输入；`action=finish` → 收敛出 `{root_cause, evidence, fix_hint}`；
- 步数上限（默认 4），到顶强制收敛；至少做一次实验再下结论。

### 3.2 为什么这才算 agent
| 流水线（原型） | 真 agent（本轮） |
|----------------|------------------|
| 一次性输出 | 多轮迭代 |
| 无工具 | 有工具（沙箱） |
| 无反馈 | 观察结果反哺下一步 |
| 不能自纠 | 假设被证伪就换假设 |

它产出的是**根因 + 实验证据 + 修复方向**，且刻意**不直接甩出整题 AC 代码**，保留教学性。
全过程 SSE 流式呈现，让人**看见它确实在多轮调用工具**——这对一个 LLM/Agent 方向的项目，是最该被看见的能力。

## 4. 数据管线与存储分层

### 4.1 数据集选型
选 **DeepMind CodeContests**：真实竞赛题 + 官方标签/难度分 + **官方测试数据**，license 干净、可离线导入，
比"爬 OJ"在合规与可得性上都更优。导入脚本 `scripts/ingest_dataset.py` 走 HuggingFace 国内镜像流式拉取，
按难度分（800–1600）与题面长度过滤，挑可判的题。

### 4.2 存储分层（一个有意识的取舍）
- **题面/元数据 → `data/problems.json`**：让 RAG 与题库下拉直接用上；
- **测试数据 → `data/tests/<id>.json` 文件**，**不进数据库**。
  原因：测试数据是只读、体量可观（百题约 400KB），放进 Neon 免费 Postgres 既浪费额度又无必要；
  作为文件随仓库/镜像分发，**线上线下判题都能用**，且零迁移负担。
- **用户态数据（账号/提交/审计/题目）→ 数据库**：需要事务、查询、按用户隔离。

一句话：**只读的内容物走文件，有状态的用户数据走数据库。**

## 5. 工程纵深的设计取舍

- **持久层**：SQLAlchemy 2.0 ORM + 连接池 + Alembic 迁移；同套代码本地 SQLite、线上 Neon Postgres，
  靠 `DATABASE_URL` 透明切换（`postgres://` 自动归一化为 `postgresql+psycopg2://`）。
- **防滥用**：分层——IP 限流（内存滑动窗口）+ 每日配额（按审计日志计数）+ 输入长度上限 + 对话话题守门。
  目的是给 LLM 花费**封顶**，防止被白嫖成通用大模型。
- **会员分级**：`credits` 算力点，`credits>0` 即 Pro。**刻意用模拟充值**——真实支付涉及商户资质与资金合规，
  学生项目以模拟积分演示"同一套付费解锁能力的工程架构"，零合规风险。
- **每日报告**：聚合当日数据 + LLM 叙述性点评，reportlab 内置中文字体导出 PDF（无外部字体依赖）。

## 6. 三个面向用户的扩展（图解 / 社群 / 分享）

这三项的目标是把系统从「能用」推向「好用、敢给人看」。每项都有一条**稳**的做法和一条**埋雷**的做法，取舍如下。

### 6.1 算法图解：预制帧，而非 LLM 现场生成
**取舍**：每个算法把输入「编译」成一串**确定性的帧**（数组状态 / 图节点状态 / DP 表格 + 当前伪代码行 + 旁白），前端三套渲染器逐帧重放——而**不是**让 LLM 现场生成动画或讲解。

**理由**
- **正确性**：可视化一旦讲错，比不讲更误导新手。预制、人工校验过的帧是确定性正确的；LLM 现场生成会幻觉；
- **零依赖**：数组用条形、图用 SVG、DP 用 `<table>`，没引任何图表/动画库，确定性、可离线、好维护；
- **可扩展**：新增算法只是再写一个「把输入变成帧数组」的纯函数，框架不动。当前覆盖排序 / 查找 / 图论 / DP 四类共 9 个。

> 这正好印证了那条尺子：在「实训要大、要美、**不能出错**」的目标下，预制版反而是对的——`不能出错` 这一轴直接否掉了 LLM 生成方案。

**与剖析联动（关键词检索而非 LLM 生成）**：分析完一道题，用一张「算法关键词 → 图解 id」的映射表去匹配题型 / 知识点 / 策略文字，命中就在剖析区给出跳转超链接。仍是**确定性匹配**——既把"学这道题"和"补这个算法"打通，又不引入 LLM 现场判断的不确定性。代价是只能链接到预制集里已有的算法（局限见 §8）。

### 6.2 社群：把 UGC 的合规风险变成一个工程点
**取舍**：社群功能做满（发帖 / 回帖 / 点赞 / 分板块 / 反馈），但发帖与回帖都先过**双层审核护栏**。

**理由**：在国内，公开 UGC 平台意味着**内容安全责任**——这是「不能出错」里最容易出事的地方。与其回避，不如把这层护栏本身做成亮点：
- **第一层 · 确定性关键词 + 长度过滤**：始终生效、不依赖网络，是真正的兜底保证（辱骂 / 广告引流 / 垃圾）；
- **第二层 · LLM 语义审核 agent**：捕捉关键词扫不到的语义违规。**best-effort，失败即放行**——因为第一层已兜住最坏情况，不让模型抖动把所有人挡在门外（可用性优先）。

这样既拿到了「覆盖面大」与「考虑了内容安全」的成熟度信号，又没引入与教学无关的合规负担。点赞用 `UNIQUE(post_id,user_id)` 约束做**去重**，避免重复刷赞。

### 6.3 分享：生成图片，而非接微信
**取舍**：「分享学习卡片」走**客户端 Canvas 出图**（保存 / 复制到剪贴板），而**不**接微信 JS-SDK。

**理由**：微信网页内的真分享需要**已认证的公众号主体**（要资质、要审核），对学生项目是不成比例的合规坑。Canvas 出的是一张**平台无关**的图片，微信 / QQ / 朋友圈都能发，零授权、零后端、确定性渲染——同时满足「美观」与「不能出错」。

**二维码为什么用本地离线库，而非二维码图片 API**：卡片要带一个指向站点的二维码做引流。若用 `api.qrserver.com` 这类外部图片接口，把图画进 Canvas 会**污染画布（tainted canvas）**，导致 `toBlob`/剪贴板写入直接抛 `SecurityError`——「保存图片 / 复制」这两个核心动作全废，且离线不可用。因此选**纯前端离线库 `qrcode-generator`**（MIT），自己算出模块矩阵后用 `fillRect` 逐格画到 Canvas：不发任何外部请求、不污染画布、导出与复制照常、断网也能出图。一个落地坑：该库在浏览器里靠顶层 `var qrcode` 暴露全局，但其 UMD 尾巴会在检测到 AMD 时走 `define()` 分支，所以脚本必须排在 Monaco 的 AMD `loader.js` **之前**加载，否则全局可能拿不到。

## 7. 安全模型

- **密钥**：绝不进仓库，本地 `.deepseek_key`（gitignore）/ 云端环境变量；
- **沙箱**：学生代码在子进程运行，环境变量经**白名单过滤**，剥离 `DEEPSEEK_API_KEY` 等，
  防止通过 `os.environ` 窃取密钥；配超时与临时工作目录。
  - **工作目录信息收敛**（补充材料2 #2）：代码文件放进隐藏的 `.src` 子目录，运行时把
    `cwd` 指向同级**空** `work` 目录，于是学生代码里 `os.getcwd()/os.listdir()` 看到的是
    一个干净空目录，不再暴露 `solution.py` 与 `cp_sandbox_` 这类沙箱命名；运行报错里的
    真实路径也统一清洗为 `main.py`。这是**信息层面**的收敛，不替代硬隔离。
  > 边界声明：这是教学用途的**轻量隔离**（子进程 + 超时 + 临时目录 + 环境过滤 + 信息收敛），
  > **未做系统调用级硬隔离**；真生产环境应换容器 / seccomp / gVisor。
- **认证**：pbkdf2-HMAC-SHA256 加盐哈希存密码，登录态用 HMAC 签名 Cookie（无状态、防篡改）。

## 8. 已知局限（诚实记录）

- **对拍是经验性的**：N 组随机数据未发现反例 ≠ 数学证明正确；暴力解本身可能写错（已用样例校验降低风险，但非零）。
- **对拍依赖小规模随机数据**：只能抓在小输入上就暴露的 bug；纯性能问题（大数据 TLE）靠沙箱超时兜底，覆盖有限。
- **沙箱非硬隔离**：见 §7，当前定位是教学/演示。
- **判题仅支持 Python**：判题与对拍管线目前面向 Python；扩展多语言需要每语言的运行/编译适配。
- **算法图解是预制集**：当前覆盖 9 个经典算法；新增算法需手写其帧生成函数，不能任意题目自动生成动画。
- **社群审核第二层 fail-open**：LLM 审核不可用时放行，最终仍由确定性关键词层兜底；单进程内存限流，多实例需换共享存储。
- **冷启动**：免费云服务休眠后首次访问约 50s 冷启动。

## 9. 补充材料2 迭代：权限分级 / 激励广告 / 题解关联（设计取舍）

这一批以「修复体验断点 + 跑通商业闭环」为目标，落地时坚持两条线：**克制的编辑感美学**、**确定性兜底优先**。

### 9.1 功能权限分级：把"免费可用"与"Pro 增值"划清
**取舍**：导师审阅、普通智能体分析对**所有人免费**；深度分析、苏格拉底导师、导师对话改为**消耗 Pro 算力点**。
**理由**：让产品有清晰的价值锚——基础能力人人可用建立信任，高耗时/高价值的对话式能力作为增值点，才撑得起"商业闭环"的叙事。后端 `_require_pro` 在 `hint/chat` 端点与 `analyze` 的 `deep` 分支兜底（非 Pro 返回 `402 PRO_REQUIRED`）；前端在入口先行拦截并弹解锁引导，**两层都拦**，避免靠前端单点。

### 9.2 模拟激励广告：演示商业模式，不接真实联盟
**取舍**：「看广告得算力点」做成**模拟激励视频**（倒计时 + 赞助卡 + 发点），而**不**接 AdSense / 百度联盟。
**理由**：真实广告联盟要 **ICP 备案 + 网站审核 + 流量门槛**，且本项目部署在境外 Render，短期无法落地；硬接只会变成不成比例的合规坑。模拟激励广告用最小代价讲清「免费用户→看广告→得点→用高级能力」的闭环，后端 `/api/ad-reward` 按**日限次**（`AD_DAILY_LIMIT`）防滥用、需登录、走审计计数。
**美学约束**：广告位刻意**不做**深色视频黑屏 + 大图标那套廉价感，而是复用暖色书桌的 `tint` 卡片 + 深青 `/A` 标记 + 衬线文案 + 一条 accent 细进度线，让它像编辑排版的一部分，不破坏原有调性。

### 9.3 题解关联：把"题"和"社群讨论"双向打通
**取舍**：发帖可**可选关联**一道题（`posts.problem_id/problem_title`，幂等迁移加列）；题目页聚合「这道题的社群讨论」，社群帖详情可一键「去做这题」。
**理由**：呼应"在每个题目中添加题解内容"的诉求，又不必为每题预置官方题解（无数据源、维护重）。让 UGC 自然沉淀到题目上，做题与复盘形成回路。关联走**确定性匹配**（题号/题名解析），不引入 LLM 现场判断。

### 9.4 题面富文本：预处理 + marked，而非纯靠 LLM
**取舍**：题面展示从纯文本升级为富文本——`【小节标题】`转加粗、样例数据转代码块、`10^9`转上标，再交 marked 渲染。
**理由**：题库题面是结构化纯文本，用一个**确定性预处理器**就能稳定美化，不必为排版去调用 LLM（省钱、零不确定性、不会把样例空格揉坏）。

### 9.5 品牌中性化
顶栏副标题去掉「— LangGraph × DeepSeek」、状态位「DEEPSEEK ONLINE」→「MODEL ONLINE」、分享卡去掉模型品牌。让产品与具体模型供应商解耦（保留 LangGraph 这类技术栈中性词）。

## 10. 复习批次：状态一致性与运营细节（设计取舍）

上线后这一轮聚焦三件事：**前端状态不自相矛盾**、**分享卡片成为引流入口**、**社群有了运营层的排序与回路**。**无新增数据库迁移**。

### 10.1 登录态：单一刷新入口，杜绝矛盾态
**问题**：曾出现「顶栏已登录、日报区却显示请先登录」。根因是日报区把后端的 401「请先登录」错误写进 DOM 后，登录态变化时没有任何地方负责把它**归位**——视图各自为政，状态四散。
**取舍**：与其在每个触发点零散地补刷新，不如收敛成**一个出口**。新增 `onAuthChanged()`，在登录 / 注册 / 退出 / 自动退出后**统一**刷新仪表盘、社群、题单、日报区；日报区单独抽出 `resetReportArea()` 按「未登录 / 非 Pro / 可生成」三态渲染正确文案，并在每次进入仪表盘时先归位。`生成今日报告` 也改为**前置**登录与 Pro 引导，不再让请求落到后端再回弹错误文案造成观感矛盾。
**理由**：状态一致性问题的根治办法是减少「谁来刷新」的不确定性——把它变成一处确定性的副作用，而非散落各处的补丁。同理，换题时用 `resetAllAgents()` 一次性把智能体团队（卡片 + 状态点 + 进度条）归零，杜绝上一题的「完成 ✓」串台。

### 10.2 闲置自动退出：放在前端的取舍
**取舍**：10 分钟无操作自动退出**做在前端**（计时器 + 交互续期），而非改服务端 Cookie 有效期。
**理由**：登录态是 HMAC 签名的无状态 token、`max_age` 30 天（方便日常使用）；「闲置即退」是一个**会话级体验诉求**，目的就是保护公共/共享设备上没关页面的账号。前端计时器能精确表达「无操作」语义且即时反馈（弹提示 + 顶栏复位），实现成本低；它不替代服务端鉴权（每个受保护端点仍各自校验 token），只是叠加的一层体验防护。

### 10.3 社群排序：默认热度，而非默认时间
**取舍**：帖子列表默认按**点赞数排序**（最热），而非最新；保留一键切「最新」。
**理由**：教学型社群里，「高赞题解 / 高赞答疑」的长期价值远大于「刚发的帖」。默认热度把沉淀下来的优质内容顶到前面，符合「来这里是为了学」的场景；同时留「最新」给想看实时动态的人。后端 `list_posts(sort=)` 统一支持 `hot`（`likes desc, created desc`）/`new`，题目页聚合也走热度口径，保持一致。

### 10.4 题解关联：把入口做到「正在做题」的地方
**取舍**：把「写本题题解」的入口**常驻在题目页**（而非只在社群里发帖时手填关联），载题即显示，无帖时也引导发第一篇。
**理由**：UGC 沉淀的关键是**降低在正确时机动手的门槛**——学生刚做完一道题、记忆最热乎时，正是写题解的最佳时刻。在题目页一键发帖并自动带上当前题目关联，比让他切到社群再手动检索题目顺手得多。配套把容错做实：关联题目若是**别人的私有自建题或已删除**，「去做这题」会给出明确提示而非静默失败（私有自建题 `U` 前缀仅作者可见，是确定性可判的边界）。

## 11. 国际化与平台内激励闭环（第四阶段）

老师建议「加英文切换、推进市场环节、用脚本批量获客、用激励政策拉真实用户」。其中**宣传/获客（Pillow 批量海报、pyautogui 跨平台批量点击私信）属于市场职责，且后者触碰平台反自动化与骚扰红线，前后端不实现**；落地的是与前后端本职相关、合法可演示的五件事。

### 11.1 全站中英切换（i18n）
**取舍**：纯前端 `i18n.js`，静态 UI 用 `data-i18n / data-i18n-ph / data-i18n-title` 标记 + 显式 key 字典，运行时按 key 替换；动态生成的界面（智能体卡片、顶栏、社群抽屉）在各 JS 里调用 `t()` / 自带 `T(zh,en)`，并监听 `langchange` 重绘。语言存 `localStorage`。
**边界**：**只翻译界面框架文案；题面、题解、LLM 输出等内容数据保持原文**——这些是用户/模型产生的中文内容，机械翻译反而失真，「切英文题目却还是中文」的预期落差需明确说明。131 个 key 均在 zh/en 双语字典中校验存在，无裸 key 泄漏。

### 11.2 平台内激励闭环（虚拟币复用算力点）
**取舍**：不另造货币体系，**直接复用已有 `billing` 算力点**做激励载体。新增：①**每日签到** `/api/checkin`（每日 1 次，按审计端点计数判重）；②**发帖/答疑奖励**——在社群 create/reply 成功后 `_reward_community` 发点，按日限次（发帖 3 次/回帖 5 次）防刷。
**理由**：把社群从「单纯讨论区」变成有正反馈的互助生态——你帮别人答疑，平台用虚拟币回报你，这是合法、可持续的「拉真实用户」方式，替代脚本刷量。所有发放都按日限次，且复用既有限流/审计，不新增滥用面。

### 11.3 用户画像面板
**取舍**：复用已有 `profile.build_profile`（档位/强弱题型/高频失误）与 `stats.type_mastery`，在仪表盘新增**可视化画像卡片**（档位徽章 + 通过率 + 擅长/薄弱题型 chips + 题型掌握度条）。画像同时驱动苏格拉底导师/对话的「因材施教」prompt 注入（既有能力）。纯展示层，零新增后端。

### 11.4 学习行为埋点：聚合，而非键鼠记录
**取舍**：老师提到「用户鼠标、键盘行为跟踪」。**全量鼠标轨迹 + 键盘记录本质是 keylogger，对辅导平台无正当用途且触碰《个人信息保护法》红线，明确不做**。落地的是**聚合学习行为**：新增 `study_logs` 表与 `/api/telemetry`，只存「专注时长 + 击键次数 + 运行/提交次数」这类**计数与时长**，绝不存任何按键内容或坐标轨迹。
**专注口径**：前端只在「页面可见 且 30s 内有过交互」时累计专注秒数，把挂机排除在外；每 30s / 切后台(`visibilitychange`) / 离开(`pagehide`, `sendBeacon`) 上报，服务端对单次上报的时长/计数**截断**（≤2h / ≤10万击键）防伪造刷量。仪表盘「学习投入」卡片展示今日/累计专注分钟、累计击键、每次提交均用时——有教育意义、可解释、且隐私安全。

### 11.5 成就徽章与结业证书
**取舍**：里程碑徽章（首次 AC / 攻克 5·20 题 / 活跃 7 天 / 高通过率 / 提交 50 次）从 `stats` **纯客户端推导**，可解释、零后端。攻克 ≥3 题解锁**结业证书**：复用分享卡的客户端 Canvas + 离线二维码技术，绘姓名/已攻克数/通过率/活跃天数/签发日期/指向站点的二维码，可保存或复制——平台无关、不接任何第三方授权，是合法的口碑裂变载体。

### 11.6 多语言判题：C++（真编译运行），不做 Java
**取舍**：编辑器 + 判题增加 **C++**，但**刻意不做 Java**。
**理由**：①C++ 是竞赛主力语言，g++ 轻量（Docker 加几十 MB）、编译快、二进制内存小，在 512MB / 0.1CPU 的免费实例上扛得住；②Java 需 JDK（镜像膨胀）且每个用例起一个 JVM，免费机内存吃力、判题会不稳，性价比差，故不支持。AI 能力（分析 / 审阅 / 提示 / 对话）本就读代码、与语言无关，三类语言天然可用。
**编译一次，多次运行**：判题会对很多用例反复运行同一份解（真值最多全量用例、对拍最多 30 组）。若每次重新 `g++` 编译，在 0.1CPU 上会慢到不可用。因此 `sandbox.Program` 把「准备(编译)」与「运行」分开：C++ 在 `prepare()` 编译一次得到二进制，judge 用同一个二进制跑完所有用例；Python 无需编译，`prepare` 只持源码。对拍里**用户解走其语言、暴力解/数据生成器始终是 LLM 产出的 Python**。
**安全**：编译出的原生二进制无法套用 Python 的 `sys.addaudithook`（那是解释器级钩子），但仍有 `setrlimit`（内存/CPU/文件大小）+ 独立进程组超时整组击杀 + 空 cwd + 不传任何密钥环境变量。与既有定位一致：教学级隔离，非系统级硬隔离（硬隔离仍需容器/seccomp）。
**实现要点**：`config.CPP_COMPILER/CPP_STD/COMPILE_TIMEOUT`；`Dockerfile` 装 `g++`（不装 JDK）；`_native_env()` 把编译器目录提到 PATH 最前（避开 Windows 多套 MinGW 的 DLL 冲突，Linux 无害）；`/api/run`、评测流、ReAct 调试 agent 全部按 `language` 分发；前端 `lang-select` 加 C++，各语言代码分桶存 `localStorage`、切换时换 Monaco 语法高亮与模板。
