# 设计说明 · ARENA 算法竞赛辅导智能体

> 本文讲的是**为什么这样设计、做了哪些取舍**，与《功能报告》（讲"有什么功能"）、README（讲"怎么用"）互补。
> 面向想理解系统骨架与工程判断的读者。

---

## 0. 设计定位：从「演示」到「可信」

一个能跑的多智能体 demo 不难，难的是让懂行的人相信它「不是玩具」。我们认为分水岭是两点，
本轮设计的全部重心都压在这两点上：

1. **判题要有「地面真值（ground truth）」。** 如果 AC 的含义只是"某个会幻觉的模型觉得你对了"，
   那整套判题就是空中楼阁。必须建立确定性的、可复现的判定来源。
2. **「Agent」要名副其实。** 真正的 agent 有工具、有反馈闭环、能自我纠错；
   而不是把几次顺序的 LLM 调用串起来就叫"多智能体"。

其余功能（题库、画像、会员、报告）都是围绕这两个支点的工程纵深。

## 1. 多智能体编排：状态机，而非一条长 prompt

**取舍**：用 LangGraph 的 `StateGraph` 把辅导拆成显式的节点与条件边，而不是写一个巨型 prompt 让模型"一口气全干"。

**理由**
- 每个角色（分析/规划/审查/判题…）有独立的系统提示词与输出契约（JSON schema），职责单一、可单测、可替换；
- **条件分支**是关键：分析流里"不是有效算法题就不硬分析"；评测流里"语法都过不了就跳过判题"。
  这些分支用图的 `add_conditional_edges` 表达，逻辑清晰且可观测；
- `graph.stream(stream_mode="updates")` 天然提供**逐节点的中间产出**，直接驱动前端"智能体逐个点亮"的可视化——
  编排结构与 UI 叙事是同构的。

**两条工作流**
```
分析流  检索 → 分析 →[是有效题?]→ 规划 / 直接结束
评测流  审查 →[语法过?]→ 双轨判题 → 汇总
```
调试回路不放进图里，而是**独立的按需 ReAct 循环**（见 §3），因为它的步数与分支由模型在运行时决定，
不是预先固定的 DAG——这本身就说明了它和"流水线"的区别。

## 2. 判题真值化（核心设计）

### 2.1 问题
competitive programming 的判题需要**确定性的期望输出**。原型阶段是"让 LLM 生成测试用例并想象期望输出"，
这有一个致命缺陷：**裁判本身会幻觉**，AC 不可信。

### 2.2 为什么不直接"接 OJ"
我们考察过 Codeforces / LeetCode / 洛谷 / AtCoder：
- **题面与元数据**（标签、难度分）多数可得；
- **但隐藏测试数据几乎都不公开**。Codeforces 有官方 API 却不给 system tests；LeetCode 判题在它自家服务器。

也就是说，"把自己做成一个真 OJ"这条路走不通——你永远缺真值。

### 2.3 双轨判题（`core/judge.py`）
按题目是否自带官方数据，自动选轨：

| 轨道 | 触发条件 | 真值来源 | 强度 |
|------|----------|----------|------|
| **真值判定 truth** | 题目带官方测试数据 | `data/tests/<id>.json` 里的真实数据 | 确定性，等同 OJ |
| **差分对拍 stress** | 用户自己粘的题、无数据 | LLM 写的**朴素暴力解** | 经验性，但能抓出绝大多数 bug |

- 真值轨：100 道导入真题全部带官方数据，逐组比对，遇首个失败即停并给反例；
- 对拍轨：让 LLM 产出 `brute_code`（朴素正确解）+ `gen_code`（小规模随机数据生成器），
  **先用题面样例校验暴力解可信度**（暴力解连样例都过不了就降可信度标注），
  再用大量小随机数据让用户解与暴力解对拍，**第一处分歧即最小反例**。

**为什么对拍是对的补丁**：它正是竞赛选手手动「对拍」的自动化版本。即便接入了 Codeforces 也拿不到隐藏数据，
对拍恰好**用一个"显然正确但慢"的解当真值**，补上了这块缺口——而且小规模随机数据天然产出**最小反例**，
直接喂给下游的调试 agent。

### 2.4 得分设计
判题裁决是铁证，**主导评分**；审查分只作 ≤5 分的风格加成：
- 真值 AC ≈ 95，对拍 AC ≈ 85（对拍是经验性结论，刻意保守）；
- 未过按通过比例给部分分（≤60），CE 封顶 20。

这样修正了原型里"正确解却被审查师的风格意见拉低分"的反直觉问题。

## 3. Agentic 调试回路（核心设计）

### 3.1 协议
`core/debugger.py` 手写了一个 **ReAct 循环**（而非黑盒地交给某框架），便于完全掌控与流式呈现：

- **唯一工具** `run(code, stdin)`：在沙箱里真实执行一段 Python，返回 stdout/stderr/状态；
- 每一步模型输出严格 JSON：`{thought, action: run|finish, code, stdin, conclusion}`；
- `action=run` → 我们执行并把观察结果作为下一轮的输入；`action=finish` → 收敛出 `{root_cause, evidence, fix_hint}`；
- 步数上限（默认 4），到顶强制收敛；至少做一次实验再下结论。

### 3.2 为什么这才算 agent
| 流水线（原型） | 真 agent（本轮） |
|----------------|------------------|
| 一次性输出 | 多轮迭代 |
| 无工具 | 有工具（沙箱） |
| 无反馈 | 观察结果反哺下一步 |
| 不能自纠 | 假设被证伪就换假设 |

它产出的是**根因 + 实验证据 + 修复方向**，且刻意**不直接甩出整题 AC 代码**，保留教学性。
全过程 SSE 流式呈现，让人**看见它确实在多轮调用工具**——这对一个 LLM/Agent 方向的项目，是最该被看见的能力。

## 4. 数据管线与存储分层

### 4.1 数据集选型
选 **DeepMind CodeContests**：真实竞赛题 + 官方标签/难度分 + **官方测试数据**，license 干净、可离线导入，
比"爬 OJ"在合规与可得性上都更优。导入脚本 `scripts/ingest_dataset.py` 走 HuggingFace 国内镜像流式拉取，
按难度分（800–1600）与题面长度过滤，挑可判的题。

### 4.2 存储分层（一个有意识的取舍）
- **题面/元数据 → `data/problems.json`**：让 RAG 与题库下拉直接用上；
- **测试数据 → `data/tests/<id>.json` 文件**，**不进数据库**。
  原因：测试数据是只读、体量可观（百题约 400KB），放进 Neon 免费 Postgres 既浪费额度又无必要；
  作为文件随仓库/镜像分发，**线上线下判题都能用**，且零迁移负担。
- **用户态数据（账号/提交/审计/题目）→ 数据库**：需要事务、查询、按用户隔离。

一句话：**只读的内容物走文件，有状态的用户数据走数据库。**

## 5. 工程纵深的设计取舍

- **持久层**：SQLAlchemy 2.0 ORM + 连接池 + Alembic 迁移；同套代码本地 SQLite、线上 Neon Postgres，
  靠 `DATABASE_URL` 透明切换（`postgres://` 自动归一化为 `postgresql+psycopg2://`）。
- **防滥用**：分层——IP 限流（内存滑动窗口）+ 每日配额（按审计日志计数）+ 输入长度上限 + 对话话题守门。
  目的是给 LLM 花费**封顶**，防止被白嫖成通用大模型。
- **会员分级**：`credits` 算力点，`credits>0` 即 Pro。**刻意用模拟充值**——真实支付涉及商户资质与资金合规，
  学生项目以模拟积分演示"同一套付费解锁能力的工程架构"，零合规风险。
- **每日报告**：聚合当日数据 + LLM 叙述性点评，reportlab 内置中文字体导出 PDF（无外部字体依赖）。

## 6. 安全模型

- **密钥**：绝不进仓库，本地 `.deepseek_key`（gitignore）/ 云端环境变量；
- **沙箱**：学生代码在子进程运行，环境变量经**白名单过滤**，剥离 `DEEPSEEK_API_KEY` 等，
  防止通过 `os.environ` 窃取密钥；配超时与临时工作目录。
  > 边界声明：这是教学用途的**轻量隔离**（子进程 + 超时 + 临时目录 + 环境过滤），
  > **未做系统调用级硬隔离**；真生产环境应换容器 / seccomp / gVisor。
- **认证**：pbkdf2-HMAC-SHA256 加盐哈希存密码，登录态用 HMAC 签名 Cookie（无状态、防篡改）。

## 7. 已知局限（诚实记录）

- **对拍是经验性的**：N 组随机数据未发现反例 ≠ 数学证明正确；暴力解本身可能写错（已用样例校验降低风险，但非零）。
- **对拍依赖小规模随机数据**：只能抓在小输入上就暴露的 bug；纯性能问题（大数据 TLE）靠沙箱超时兜底，覆盖有限。
- **沙箱非硬隔离**：见 §6，当前定位是教学/演示。
- **判题仅支持 Python**：判题与对拍管线目前面向 Python；扩展多语言需要每语言的运行/编译适配。
- **冷启动**：免费云服务休眠后首次访问约 50s 冷启动。
