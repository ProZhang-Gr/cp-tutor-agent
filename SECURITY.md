## 一、概述

本系统面向公网公测，最大的攻击面是「**在沙箱里执行访客提交的任意 Python 代码**」，
其次是「**未授权/无限流的高危端点**」与「**LLM 调用花费被刷爆**」。

本次审计覆盖：代码执行沙箱、认证与口令、鉴权与越权（IDOR）、注入（SQL/路径/提示词）、
XSS/CSRF、限流与配额、密钥管理、信息泄露、滥用与成本兜底、监控可观测性。

**结论**：架构基本盘稳健（ORM 参数化、口令加盐哈希、HMAC 签名 Cookie、SameSite=Lax、
密钥不进仓库、沙箱剥离环境变量）。审计发现 **5 个公测阻断级问题，已全部修复并测试**；
另有若干中低危项，部分已加固、部分列为「待决/已知边界」。

## 二、威胁模型

| 资产 | 主要威胁 | 处置 |
|------|----------|------|
| 服务器 / 云实例 | 经沙箱代码 RCE、外发网络当攻击跳板、资源耗尽 DoS | 沙箱加固（见 §3.1）+ 端点限流 |
| 用户口令 | 库泄露后离线爆破、明文泄露、暴力破解、枚举 | 强哈希 + 限流 + 时间抹平（§3.2） |
| LLM 花费（DeepSeek） | 自助充值/伪造 IP 刷量、未授权端点白嫖 | 多层配额 + 全站日上限（§3.3） |
| 用户数据（提交/题目/口令哈希） | 越权读取、沙箱内读库、目录穿越 | 按用户隔离 + 沙箱禁读应用目录 + 路径校验 |
| UGC（社群） | 存储型 XSS、违规内容 | 输出转义 + 双层审核护栏 |
| 管理后台 `/admin` | 弱口令爆破、越权访问、CSRF、敏感操作无审计 | 独立 HMAC 管理态（12h 失效）+ 失败锁定 + 全部数据端点鉴权守门 + 登录审计（§3.5） |

## 四、已实施的加固（细节）

### 3.1 代码执行沙箱（`core/sandbox.py`）—— 分层纵深
1. **安全审计钩子**：向子进程注入 `sitecustomize.py` + `sys.addaudithook`，**禁止**：
   - 网络（`socket.connect/bind/getaddrinfo`、`urllib`、`ftplib`、`smtplib`…）；
   - 创建子进程（`subprocess.Popen`、`os.system`、`os.exec*`…）；
   - 读写**应用目录**（口令哈希库 `progress.db`、测试数据、源码）——临时沙箱目录与标准库不受影响。
2. **资源上限（POSIX `setrlimit`）**：地址空间 320MB（挡内存炸弹）、CPU 时间、单文件写入 16MB（挡磁盘填充）。
3. **进程组隔离**：`start_new_session` + 超时整组 `killpg`，清理 fork 出来的子孙（挡 fork 炸弹）。
4. **临时目录清理**：`finally` 中 `rmtree`，杜绝磁盘泄漏堆积。

> 已测：正常解题 + 标准库 + stdin 照常；网络 / 读源码 / 起子进程**全部被拦**。

### 3.2 认证与口令（`core/auth.py`）
- pbkdf2-HMAC-SHA256 迭代 **100k → 260k**，哈希**自描述**存迭代次数；旧账号兼容校验、登录成功**自动升级**。
- 口令上限 **128 位**（防超长 pbkdf2 CPU DoS）；最短 **6→8 位**（仅约束新注册）。
- 用户不存在时也跑一次等强度哈希，**抹平时间差防用户名枚举**。
- 既有正确实践：加盐、不存明文、`hmac.compare_digest` 常量时间比较、HTTPS 下 `Secure` Cookie、`HttpOnly`。

### 3.3 防滥用与成本兜底（`core/guard.py` · `app.py` · `config.py`）
- **全站每日 LLM 总上限** `GLOBAL_DAILY_LLM_CAP`（默认 3000，环境变量可调，**含 Pro**）：到顶全员当日降级，给 DeepSeek 花费硬兜底。
- 修复**扣费时序 bug**：原来 Pro 被限流拦下仍白扣算力点，改为**通过后才扣点**。
- 叠加既有：每 IP 20 次/分限流、游客 30 / 普通用户 100 每日配额、充值封顶、输入限长、对话话题守门。

### 3.4 可观测 / IP 监控
- 每次 AI 调用落库 `audit_log`：时间 + 用户 + **IP** + 端点；meta 额外记**真实 socket `peer`、完整 `xff` 链、`ua`**。
- 只读运维工具 **`scripts/monitor.py`**（不开 HTTP 端点）：近 N 小时总量/占上限比、Top IP、Top 账号、端点分布、可疑信号（单 IP 刷量 / 单账号多 IP）与最近明细。线上连同一 `DATABASE_URL` 即可查。

### 3.5 管理后台 / 找回密码（`core/admin.py` · `core/support.py` · `app.py`）—— 2026-06-24 增补
- **鉴权隔离**：管理态用独立 HMAC 签名 token（`admin:<签发时间>.<sig>`，**12h 自动失效**），与普通用户登录态 Cookie（`arena_session`）完全分离，互不影响；管理 Cookie `arena_admin` 为 `HttpOnly` + HTTPS 下 `Secure` + `SameSite=Lax`。口令用 `hmac.compare_digest` 常数时间比对。
- **全部数据端点守门**：`/api/admin/*` 每个端点先过 `_require_admin`（token 无效即 401），`/admin` 页面本身只是静态 HTML、闸门在客户端，**真正的数据从不在未鉴权时下发**。
- **爆破锁定**（独立于全站 20/分限流，更严格）：按 IP 记登录失败时间戳，10 分钟内失败 **6 次即锁定约 10 分钟**；成功登录清零。
- **登录审计**：成功 `admin_login` / 失败 `admin_login_fail` 均落 `audit_log`，`monitor.py` 可见，便于发现穷举。
- **默认弱口令告警**：启动时若仍是 `manager/123456` 则打印安全警告（见 S6）。
- **密码不可逆红线**：后台用户列表的「密码」列**只展示哈希指纹**（算法 + 迭代次数 + hash/salt 前缀），系统与管理员都**无法**还原明文（pbkdf2 单向）；接管账号只能走「重置为新口令」。
- **找回密码不暴露存在性**：`/api/password-reset` 对存在与否的用户名都回同一成功文案，仅在账号真实存在时落工单（同名待处理工单 ≤3，防刷）；管理员核实身份后写新哈希。
- **越权 / 注入面**：用户增删改查全走 ORM 参数化；`delete_user` 删私有数据、社群帖子改匿名（保引用完整性）；后台前端对 `username/contact/note/meta` 等全部 `esc()` 转义，无存储型 XSS；状态变更端点为 `POST/PATCH/DELETE` + `SameSite=Lax` Cookie，挡住跨站 CSRF。

## 五、残余风险与边界声明

- **沙箱是「教学级」进程内隔离，非系统级硬隔离**。已挡住绝大多数滥用，但理论上仍存在 CPython 审计钩子绕过的可能。要做到真正可信，需容器 / gVisor / seccomp。**公测可接受，勿当作可跑恶意代码的生产级 OJ。**
- **限流/配额基于内存 + 单实例**：进程重启清零、多实例不共享；当前 Render 免费单实例可用，水平扩容需换共享存储（Redis）。
- **XFF 可伪造**：用于识别的 `ip` 字段可被绕过；成本侧由全站日上限兜底，取证侧由 `peer/xff` 原始记录兜底。
